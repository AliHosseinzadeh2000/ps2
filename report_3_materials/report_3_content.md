## ۱. مقدمه و انگیزه

در گزارش‌های قبلی، معماری کلی ربات آربیتراژ و موتور اصلی معامله‌گری پیاده‌سازی شد. ربات قادر است فرصت‌های آربیتراژ را بین صرافی‌های ایرانی (نوبیتکس، والکس، اینوکس) شناسایی و معاملات را به‌صورت همزمان اجرا کند.

در این مرحله، یک لایه هوش مصنوعی به سیستم اضافه شده است که هدف آن **کاهش کارمزد معاملات** از طریق انتخاب هوشمند نوع سفارش است.

### مسئله: دو نوع سفارش با کارمزد متفاوت

در صرافی‌ها، دو نوع اصلی سفارش وجود دارد:

| نوع سفارش | توضیح | کارمزد | ریسک |
|-----------|-------|--------|------|
| **Taker** | سفارش limit در بهترین قیمت موجود — فوری اجرا می‌شود | ۰.۲۵٪ | بدون ریسک — همیشه پر می‌شود |
| **Maker** | سفارش limit کمی دورتر از بهترین قیمت — در دفتر سفارش می‌ماند | ۰.۱۰٪ | ممکن است پر نشود اگر بازار تغییر کند |

**نکته مهم:** هر دو نوع سفارش، از نوع *limit* هستند (نه market). تفاوت تنها در قیمت گذاشته‌شده است:
- Taker: قیمت برابر بهترین عرضه/تقاضا → فوری اجرا می‌شود
- Maker: قیمت کمی پایین‌تر (برای خرید) یا کمی بالاتر (برای فروش) → در orderbook می‌ماند

**انگیزه اقتصادی:**  
با حجم معاملاتی ۱۰۰ میلیون تومان در روز:
- همیشه Taker: ۲۵۰,۰۰۰ تومان کارمزد روزانه
- همیشه Maker: ۱۰۰,۰۰۰ تومان کارمزد روزانه (در صورت پر شدن)

اما «همیشه Maker» خطرناک است — اگر بازار نوسان داشته باشد، سفارش پر نمی‌شود و فرصت آربیتراژ از دست می‌رود. **راه‌حل: یاد دادن به سیستم برای تشخیص زمانی که بازار آرام‌تر است و Maker امن‌تر است.**

## ۲. رویکرد: طبقه‌بندی با XGBoost

### چرا XGBoost؟

مدل XGBoost (Extreme Gradient Boosting) برای این مسئله انتخاب شد به دلایل زیر:

- **بهترین الگوریتم برای داده‌های جدولی:** در مسابقات Kaggle و تحقیقات مالی ثابت شده که برای داده‌های ساختاریافته از شبکه‌های عصبی بهتر عمل می‌کند
- **استنتاج سریع:** هر پیش‌بینی در کمتر از ۱ میلی‌ثانیه — مناسب برای معامله‌گری لحظه‌ای
- **قابل توضیح:** feature importance نشان می‌دهد چرا مدل هر تصمیمی گرفته
- **نیاز به داده کمتر:** با چند صد نمونه آموزش می‌بیند — مناسب برای بازارهای ایرانی با نقدینگی محدود

### مسئله به‌صورت رسمی

**ورودی:** ۱۹ ویژگی استخراج‌شده از orderbook لحظه‌ای  
**خروجی:** `1` (Maker امن است) یا `0` (باید Taker استفاده شود)  
**آموزش:** از روی داده‌های تاریخی orderbook با برچسب‌گذاری خودکار

## ۳. استخراج ویژگی (Feature Engineering)

### ۱۹ ویژگی orderbook

برای هر لحظه از بازار، ۱۹ ویژگی از orderbook استخراج می‌شود:

```
ویژگی‌های قیمت:
  best_bid        — بهترین قیمت تقاضا (خریدار)
  best_ask        — بهترین قیمت عرضه (فروشنده)
  mid_price       — میانگین bid و ask
  spread          — فاصله bid-ask (به تومان)
  spread_percent  — فاصله bid-ask (به درصد) ← مهم‌ترین ویژگی

ویژگی‌های عمق بازار:
  bid_depth       — حجم کل سفارش‌های خرید در orderbook
  ask_depth       — حجم کل سفارش‌های فروش در orderbook
  bid_price_levels  — تعداد سطوح قیمتی در سمت خرید
  ask_price_levels  — تعداد سطوح قیمتی در سمت فروش
  depth_imbalance   — عدم تعادل خرید/فروش (مثبت = فشار خرید)

ویژگی‌های VWAP (Volume-Weighted Average Price):
  bid_vwap        — میانگین قیمت وزن‌دار خرید
  ask_vwap        — میانگین قیمت وزن‌دار فروش
  bid_price_std   — انحراف معیار قیمت‌های خرید
  ask_price_std   — انحراف معیار قیمت‌های فروش

ویژگی‌های فشار بازار:
  buy_pressure    — نسبت حجم خرید به کل حجم
  sell_pressure   — نسبت حجم فروش به کل حجم
  buy_pressure_10  — فشار خرید در ۱۰ سطح اول
  sell_pressure_10 — فشار فروش در ۱۰ سطح اول
  pressure_ratio   — نسبت buy_pressure به sell_pressure
```

این ویژگی‌ها بر اساس ادبیات «market microstructure» (ریزساختار بازار) انتخاب شده‌اند، به‌ویژه کارهای Cont و Stoikov در حوزه پیش‌بینی حرکت قیمت از روی orderbook.

### استراتژی برچسب‌گذاری تطبیقی

**چالش:** چطور برچسب `maker` یا `taker` به هر لحظه اختصاص دهیم؟

**روش ساده (رد شد):** اگر قیمت در ۵ ثانیه آینده بیش از X تغییر کرد → taker  
**مشکل:** مقدار ثابت X باعث عدم تعادل شدید کلاس می‌شود (۹۰٪ maker / ۱۰٪ taker)

**روش تطبیقی (استفاده شد):**

```python
# محاسبه امتیاز نوسان برای هر لحظه
volatility_score = (price_change_pct / spread_pct) + max(0, spread_change_ratio - 1.0)

# استفاده از میانه به‌عنوان آستانه (نه مقدار ثابت)
threshold = median(all_volatility_scores)

label = 1 (maker)  اگر score <= threshold  # بازار آرام
label = 0 (taker)  اگر score > threshold   # بازار نوسانی
```

**نتیجه:** توزیع متعادل (~50٪ maker / 50٪ taker) که از bias در آموزش جلوگیری می‌کند.

## ۶. نتایج ارزیابی

### معیارهای اندازه‌گیری

(خروجی فایل `03_model_evaluation.txt` را اینجا paste کنید)

### تفسیر نتایج

**دقت ۶۴٪ — چرا کافی است؟**

در پیش‌بینی رویدادهای ریزساختار بازار در بازه‌های زمانی کوتاه (۴-۵ ثانیه)، حتی شرکت‌های HFT (High-Frequency Trading) با میلیاردها دلار سرمایه و تیم‌های تحقیقاتی بزرگ نمی‌توانند از دقت ~۸۰٪ عبور کنند. دقت ۶۴٪ یک نتیجه **واقعی و قابل دفاع** است.

**ROC-AUC چیست؟**

ROC-AUC معیاری است که نشان می‌دهد مدل چقدر قادر است دو کلاس را از هم تفکیک کند:
- `0.5` = تصادفی (هیچ قدرت پیش‌بینی ندارد)
- `0.689` = مدل ما (قدرت پیش‌بینی واقعی)
- `1.0` = کامل

**بهبود بین تکرارها:**
- تکرار ۱: ROC-AUC = 0.547 (+9.4٪ از شانس)
- تکرار ۲: ROC-AUC = 0.689 (+37.8٪ از شانس) — **بهبود ۲۶٪**

این بهبود نشان می‌دهد داده متنوع‌تر (نه لزوماً بیشتر) کلید اصلی است.

**ماتریس آشفتگی (Confusion Matrix):**

```
                    پیش‌بینی مدل
                  Taker   Maker
واقعیت Taker  [   157     108 ]  ← ۵۹٪ تشخیص درست
واقعیت Maker  [    69     158 ]  ← ۷۰٪ تشخیص درست
```

مدل برای Maker دقت بهتری دارد که مطلوب است — اشتباه در تشخیص Maker (false positive) هزینه بیشتری دارد.

### اعتبارسنجی متقاطع (Cross-Validation)

برای اطمینان از اینکه مدل overfitting نکرده، اعتبارسنجی ۵-fold انجام شد:

```
دقت:    ۶۲.۸٪ ± ۳.۹٪
F1:     ۶۱.۸٪ ± ۴.۰٪
ROC-AUC: 0.691 ± 0.031
```

انحراف معیار پایین (±۳.۹٪) نشان می‌دهد مدل پایدار است و نتایج به‌طور تصادفی نیست.

## ۸. یکپارچه‌سازی با ربات

### معماری کلی

```
Price Stream
    │
    ▼
Orderbook (Nobitex, Wallex, Invex)
    │
    ├──► Arbitrage Engine ──► فرصت شناسایی شد
    │
    └──► AI Predictor ──────► maker یا taker؟
              │
              ▼
         Order Executor ──► سفارش با قیمت مناسب ثبت می‌شود
```

### نکته مهم: بدون overhead اضافی

یکی از نگرانی‌های اصلی این بود که آیا AI درخواست‌های اضافی به API صرافی‌ها می‌فرستد یا نه. پاسخ: **خیر.**

سیستم از همان orderbook که برای شناسایی فرصت آربیتراژ دریافت شده استفاده می‌کند. هیچ درخواست شبکه‌ای اضافه‌ای وجود ندارد. تأخیر AI: کمتر از ۱ میلی‌ثانیه (استنتاج محلی XGBoost).

### کنترل AI

```bash
# فعال/غیرفعال کردن AI بدون تغییر کد
AI_ENABLED=true    # پیش‌فرض — AI تصمیم می‌گیرد
AI_ENABLED=false   # همیشه Taker (حالت امن)

# تنظیم buffer قیمت برای سفارش Maker
MAKER_PRICE_BUFFER_PERCENT=0.05
```

### منطق تصمیم‌گیری

```python
if use_maker is not None:
    # override دستی از API
    use_maker = use_maker
elif AI_ENABLED and predictor.is_ready():
    # AI تصمیم می‌گیرد
    is_maker, probability = predictor.predict_from_orderbook(orderbook)
    if is_maker:
        price = best_price * (1 - 0.0005)  # buffer برای ماندن در orderbook
else:
    # fallback ایمن
    use_maker = False  # همیشه Taker
```

### رفتار در خطا

اگر AI به هر دلیلی fail شود (مدل بارگذاری نشده، خطای شبکه، etc.)، سیستم **به‌صورت خودکار** به حالت Taker برمی‌گردد. ربات بدون AI هم کار می‌کند.
